Como utilizar Git e Github - Um guia prático e completo

Este documento serve como um guia completo para quem está iniciando no Git. Cada comando ou função está descrito fora da área cinza, enquanto a área cinza contém apenas o comando em si. Ao final de algumas seções, listamos as principais flags disponíveis para tornar o uso dos comandos ainda mais poderoso.

1. Instalação do Git

Windows

Opção 1 - Download direto:

  1. Baixar o instalador em: git-scm.com/download/win
  2. Executar o arquivo .exe baixado
  3. Manter as opções padrão no instalador (recomendado)

Opção 2 - Via PowerShell (Administrador):

Instalar com winget:

O winget é um gerenciador de pacotes oficial para Windows. Basta rodar esse comando em um terminal com privilégios de administrador para instalar o Git. 

winget install --id Git.Git -e --source winget
OU com Chocolatey:

Chocolatey é outro gerenciador de pacotes para Windows. Caso já tenha instalado, você pode usar este comando para instalar o Git rapidamente. 

choco install git -y

Linux (Ubuntu 22.04+)

1. Atualizar repositórios:

Atualiza a lista de pacotes disponíveis em seu sistema Ubuntu, garantindo a versão mais recente do Git. 

sudo apt update
2. Instalar dependências:

Instala utilitários necessários para gerenciar pacotes e possíveis repositórios. Pode não ser obrigatório em todas as distros, mas é recomendado. 

sudo apt install software-properties-common
3. Instalar Git:

Finalmente, instala o Git confirmando automaticamente com a flag -y. Agora você já terá o Git configurado no sistema. 

sudo apt install git -y

Verificação da Instalação

Verificar versão do Git:

Exibe a versão do Git instalada. Se retornar algo como git version 2.X.X, significa que a instalação foi bem-sucedida. 

git --version

Links Úteis:

  • Download Git: git-scm.com/downloads
  • Documentação: git-scm.com/doc
  • Guia Git: git-scm.com/book/pt-br/v2

2. Configuração Inicial

Definir nome de usuário global:

Todo commit que você fizer será assinado com este nome, aparecendo no histórico de commits. 

git config --global user.name "Seu Nome"
Definir email global:

Assim como o nome, o email aparecerá no histórico de commits para identificar o autor de cada mudança. 

git config --global user.email "seu@email.com"

3. Comandos Básicos

Criar um novo repositório Git:

Inicia o repositório Git no diretório atual. Cria uma pasta oculta .git onde fica todo o controle de versão. 

git init
Verificar o status das alterações:

Mostra quais arquivos foram modificados ou adicionados e quais estão prontos para commit. É um comando muito usado antes de cada commit. 

git status

4. Branches

Listar todas as branches:

Exibe todas as branches (ramificações) disponíveis no repositório local, indicando a branch atual com um asterisco (*). 

git branch
Criar e mudar para nova branch:

Cria uma branch chamada nova-branch e já posiciona o HEAD dentro dela. Útil para começar a desenvolver uma nova feature separadamente. 

git checkout -b nova-branch
Mudar para branch existente:

Troca para a branch nome-branch se ela já existir. Ideal para continuar o desenvolvimento ou revisar código. 

git checkout nome-branch

5. Staging e Commit

Adicionar todos os arquivos ao staging:

Adiciona todas as alterações (novos arquivos, modificações, deleções) ao “staging area”. É a etapa antes de criar um commit. 

git add .
Criar um commit com mensagem:

Registra as mudanças adicionadas no staging. A mensagem deve descrever o que foi feito, para facilitar o histórico. 

git commit -m "mensagem do commit"
Criar commit com “add” automático:

Você pode pular o git add usando -a junto com -m. Isso faz com que o Git automaticamente “adicione” (staging) todos os arquivos modificados que já estão sendo rastreados e, em seguida, crie o commit com a mensagem especificada: 

git commit -am "sua mensagem de commit"

Flags Principais do git commit

  • -m "mensagem": Define a mensagem do commit diretamente na linha de comando.
  • -a: Adiciona automaticamente arquivos modificados ao commit, sem precisar git add.
  • -am: Combina as opções -a e -m numa só.
  • --amend: Modifica o último commit (útil para corrigir mensagens ou mudanças esquecidas).
  • --no-edit: Usado com --amend para manter a mesma mensagem de commit.
  • --allow-empty: Permite criar um commit mesmo que não haja alterações.
  • -v ou --verbose: Mostra o diff das alterações que serão commitadas.
  • -q ou --quiet: Oculta a maioria das mensagens do commit.
  • -S ou --gpg-sign: Assina o commit com GPG.

6. Criação de Branches

Branches são uma parte fundamental do Git:

Elas permitem que você trabalhe em diferentes partes do projeto sem interferir no código principal. Cada branch é uma linha de desenvolvimento separada, que pode ser mesclada de volta à main quando estiver pronta.

Nesta seção, veremos como listar branches, criar novas, mudar entre elas e excluí-las.

Ver todas as branches existentes:

Exibe a lista de branches locais. A branch atual estará em destaque. 

git branch
Criar nova branch (sem mudar para ela):

Se você não quer “entrar” imediatamente na nova branch, pode criá-la separadamente. 

git branch nova-branch
Criar e entrar na nova branch:

Faz ao mesmo tempo a criação da branch e a troca para ela. Bem prático. 

git checkout -b nova-branch
Mudar para uma branch existente:

Volta para uma branch criada anteriormente, caso queira continuar o trabalho ali. 

git checkout nome-branch
Deletar uma branch local:

Remove a branch localmente. É importante que ela não tenha commits pendentes que não foram mesclados. 

git branch -d nome-branch
Deletar uma branch remota:

Remove a branch do servidor remoto, limpando branches que não serão mais utilizadas. 

git push origin --delete nome-branch

7. Fluxo de Trabalho Básico

Exemplo de fluxo desde o início:

1. Criar nova branch a partir da main:

É boa prática criar sempre novas branches para features ou correções, evitando trabalhar diretamente na main. 

git checkout -b minha-branch
2. Ver status das alterações:

Para verificar quais arquivos foram modificados ou adicionados desde o último commit. 

git status
3. Adicionar arquivos alterados:

Deixa tudo pronto para ser commitado (staged). 

git add .
4. Criar commit:

Grava o histórico das mudanças localmente, com uma mensagem que descreve o que foi feito. 

git commit -m "meu primeiro commit"
5. Voltar para main:

Normalmente voltamos à main para trazer atualizações ou mesclar a branch. 

git checkout main
6. Atualizar main:

Caso existam alterações remotas, este comando atualiza a branch local. 

git pull
7. Juntar alterações da branch:

Mescla as mudanças de minha-branch na main. 

git merge minha-branch
8. Enviar para o GitHub:

Publica as alterações no repositório remoto, acessível para o restante da equipe. 

git push

8. Trabalho Remoto

Adicionar um repositório remoto:

Associa o seu repositório local a um repositório remoto chamado origin. Permite enviar e receber alterações via internet. 

git remote add origin https://github.com/usuario/repositorio.git
Enviar alterações para o repositório remoto:

Faz o “upload” das alterações da sua branch local main para o repositório remoto (nomeado como origin). 

git push origin main
Atualizar seu repositório local:

Faz o “download” das alterações que possam estar no remoto e mescla com sua main local. 

git pull origin main

Flags Principais do git remote

  • git remote -v: Lista todos os repositórios remotos e suas URLs (fetch e push).
  • git remote add <nome> <url>: Cria um repositório remoto localmente (por ex.: origin).
  • git remote remove <nome>: Remove o repositório remoto configurado.
  • git remote rename <old> <new>: Renomeia o repositório remoto.
  • git remote set-url <nome> <url>: Altera a URL de um repositório remoto existente.
  • git remote set-head <nome> <branch>: Define qual é a branch principal de um remoto.

Não existem muitas flags específicas em git remote, mas sim subcomandos (add, remove, rename, etc.). origin é apenas o nome padrão que damos ao repositório remoto principal.

9. Tags

Criar uma nova tag:

Marca um ponto específico da sua linha do tempo do Git, geralmente usado para marcar versões de lançamento. 

git tag -a v1.0 -m "versão 1.0"
Enviar tags para o repositório remoto:

Envia todas as tags que você criou localmente para o repositório remoto. 

git push origin --tags

Flags Principais do git tag

  • -a ou --annotate: Cria uma tag anotada (armazena nome, email, data, mensagem).
  • -m "mensagem": Define a mensagem da tag (tags anotadas).
  • -s ou --sign: Assina a tag usando GPG.
  • -u <chave> ou --local-user <chave>: Usa uma chave GPG específica.
  • -f ou --force: Sobrescreve (força) a criação de uma tag existente.
  • -d ou --delete: Deleta uma tag local.
  • -l ou --list: Lista todas as tags do repositório.
  • -n<num>: Mostra <num> linhas da mensagem de cada tag listada.

10. Versionamento Semântico

Estrutura de Versão: MAJOR.MINOR.PATCH

  • MAJOR (1.0.0): Mudanças incompatíveis
  • MINOR (1.1.0): Novas funcionalidades compatíveis
  • PATCH (1.0.1): Correções de bugs

O versionamento semântico ajuda a entender rapidamente o que mudou em uma nova versão: se houve grandes quebras de compatibilidade ou apenas melhorias pontuais.

Fluxo de Versionamento

Exemplo de evolução de versões:

v1.0.0 - Lançamento inicial
├── v1.0.1 - Correção de bugs
├── v1.1.0 - Nova funcionalidade
│   └── v1.1.1 - Correção na funcionalidade
└── v2.0.0 - Breaking changes

11. Reversão de Versões

Métodos disponíveis (do mais seguro ao mais perigoso):

  1. git revert: cria um novo commit que desfaz alterações.
  2. git checkout: navega entre versões (modo leitura, não altera histórico).
  3. git reset: reescreve o histórico. Use com cautela.

Sempre que precisar “voltar atrás” em algo que já foi commitado, planeje com cuidado qual método é o mais apropriado para não impactar o fluxo da equipe.

1. Ver histórico de commits:

Lista os commits de forma resumida (ID e mensagem). É muito útil para localizar o commit exato que deseja reverter. 

git log --oneline
2. Navegar para tag específica (modo leitura):

Faz o checkout de um tag, permitindo que você veja o código como estava nessa versão, mas sem alterar o histórico principal. 

git checkout v1.0.0
3. Voltar para branch principal após visualização:

Retorna para sua branch de trabalho (geralmente main) depois de ter “navegado” para outro commit/tag. 

git checkout main
4. Criar branch de segurança:

Antes de fazer revert ou reset, crie uma branch para não mexer diretamente na main. Assim, você evita impactar todo o histórico. 

git checkout -b rollback-feature
5. Reverter para commit específico:

Cria um novo commit que desfaz as alterações do commit b8c9d14. Preserva o histórico, ideal para equipes. 

git revert b8c9d14
6. Reverter o último commit (HEAD):

Desfaz o commit mais recente. Se você acabou de subir algo errado, este comando é uma forma rápida de corrigir. 

git revert HEAD

⚠️ Importante:

  • Sempre crie uma branch antes de reversões grandes;
  • Use git revert ao invés de git reset em branches compartilhadas;
  • Faça backup antes de operações destrutivas;
  • Comunique a equipe sobre reversões em produção.

Exemplo de Fluxo Completo de Reversão:

1. Verificar tags disponíveis:
git tag
2. Ver histórico de commits:
git log --oneline

Reversão por Tag:

3a. Criar branch de segurança e voltar para tag:
git checkout -b rollback-v1
git checkout v1.0.0

Reversão por Commit:

3b. Criar branch e reverter para commit específico:
git checkout -b rollback-commit
git revert a7d3f12
4. Se aprovado, merge com main:
git checkout main
git merge rollback-v1

12. Ferramentas Úteis

Mostrar o histórico de commits:

Mostra todo o histórico de commits em detalhes (autor, data, mensagem). Com opções adicionais, é possível ver o diff de cada commit. 

git log
Mostrar diferenças entre arquivos:

Compara o estado atual dos arquivos com o último commit ou entre dois commits específicos, exibindo as alterações linha a linha. 

git diff

13. Resolução de Conflitos

Verificar arquivos em conflito:

Exibe quais arquivos estão em conflito após um merge. Você precisa editar manualmente esses arquivos para decidir o que fica. 

git status

Passos para resolver conflitos:

  1. Abrir arquivos marcados como conflitantes.
  2. Procurar pelas marcações <<<<<<<, ======= e >>>>>>>.
  3. Escolher as alterações que devem permanecer (ou mesclar as duas partes).
  4. Remover as marcações de conflito e salvar o arquivo.
  5. Adicionar e commitar as alterações resolvidas.

14. Git Stash

Salvar alterações temporariamente com descrição:

Move as alterações atuais para uma “pilha” (stash) e limpa seu ambiente de trabalho, sem precisar fazer commit. 

git stash save "implementando nova feature de login"
Listar stashes salvos:

Mostra todos os “rascunhos” salvos pelo stash, cada um com um índice (stash@{0}, stash@{1}, etc.). 

git stash list
Aplicar stash específico:

Aplica as alterações do stash escolhido ao seu diretório de trabalho, mas mantém o stash salvo na lista. 

git stash apply stash@{0}
Recuperar e remover o último stash:

Aplica as alterações do stash mais recente e o remove da lista. 

git stash pop

Exemplo de fluxo com stash:

# Salvando trabalho em progresso
git stash save "feature login incompleta"

# Mudando de branch para correção urgente
git checkout hotfix

# Fazendo correção...
git commit -m "fix: corrige erro crítico"

# Voltando à branch original
git checkout feature
git stash pop  # Recupera alterações salvas do stash e as remove da pilha

15. Git Ignore

Arquivo .gitignore:

# Arquivos compilados
*.class
*.jar

# Diretórios
node_modules/
dist/
build/

# Arquivos de ambiente
.env
.env.local

# Logs e temporários
*.log
temp/

O arquivo .gitignore informa ao Git quais arquivos ou pastas devem ser ignorados (não rastreados). Ideal para não enviar credenciais ou arquivos grandes.

16. Boas Práticas de Commit

Estrutura de mensagem de commit:

tipo(escopo): resumo curto

descrição detalhada

Exemplos práticos com comandos:

# Nova funcionalidade
git add src/auth/*
git commit -m "feat(auth): adiciona autenticação via Google

Implementa OAuth2 com Google
- Adiciona botão de login
- Configura rotas de callback
- Adiciona middleware de autenticação"

# Correção de bug
git add src/login/validator.js
git commit -m "fix(login): corrige validação de senha

Resolve o problema de comparação de senhas que causava
falhas no login quando usados caracteres especiais"

# Documentação
git add README.md
git commit -m "docs(readme): atualiza instruções de instalação

- Adiciona passo a passo de configuração
- Atualiza requisitos mínimos
- Inclui exemplos de uso"

# Refatoração
git add src/database/*
git commit -m "refactor(database): migra queries para prepared statements

- Melhora segurança contra SQL injection
- Otimiza performance das consultas
- Remove código duplicado"

# Testes
git add tests/api/*
git commit -m "test(api): adiciona testes para endpoints de usuário

- Adiciona testes de integração
- Cobre casos de erro
- Implementa mocks para dependências externas"

Padronizar as mensagens de commit facilita o entendimento do histórico e a colaboração com a equipe. Use tipo para indicar se é uma nova feature (feat), correção (fix), documentação (docs), etc.

Fluxo de Trabalho Completo com Git

A seguir, veremos um fluxo prático inspirado no Git Flow (aquele que envolve as branches main, develop e possivelmente release/hotfix) para organizar melhor o processo de desenvolvimento.

Cenário: Você está começando um novo projeto...

1. Criar ou Clonar Repositório e Criar Branch develop

Se você ainda não tem um repositório criado no GitHub/GitLab... 

git clone https://github.com/usuario/novo-projeto.git
git init
git checkout -b develop
2. Configurar Git (Nome e Email)

Mesmo após clonar ou iniciar um repositório, lembre-se de definir... 

git config --global user.name "Seu Nome"
git config --global user.email "email@exemplo.com"
3. Criar Feature Branch

Para cada nova funcionalidade (feature), crie uma branch a partir da develop. 

git checkout develop
git checkout -b feature-api
4. Fazer Alterações e Commits na Feature

Edite seu projeto e então use git add seguido de git commit. 

git add .
git commit -m "feat(api): implementa endpoints iniciais"
5. Mesclar Feature na develop

Após concluir o desenvolvimento da feature... 

git checkout develop
git pull
git merge feature-api
6. Iniciar uma Release Branch

Quando chegar a hora de preparar um lançamento oficial... 

git checkout -b release-1.0.0 develop

Após testar e garantir que a release-1.0.0 está estável... 

git checkout main
git merge release-1.0.0
git checkout develop
git merge release-1.0.0
7. Criar Tag de Versão

Agora que a main está pronta para produção... 

git checkout main
git tag -a v1.0.0 -m "Lançamento oficial da versão 1.0.0"
8. Enviar Tudo para o Remoto

Agora, envie as alterações da main e as novas tags para o repositório remoto. 

git push origin main
git push origin --tags
9. Correção Urgente via Hotfix Branch

Se você encontrar um problema grave na produção (ou seja, na main)... 

git checkout -b hotfix-1.0.1 main
git add .
git commit -m "fix: corrigir erro crítico em produção"
git checkout main
git merge hotfix-1.0.1
git checkout develop
git merge hotfix-1.0.1
10. Tag da Correção e push

Marque a correção com uma nova tag (ex.: v1.0.1) e envie para o remoto. 

git checkout main
git tag -a v1.0.1 -m "Hotfix versão 1.0.1"
git push origin main
git push origin --tags

Flags Principais do git push

  • -u ou --set-upstream: Define a branch remota de acompanhamento.
  • -f ou --force: Força o push, sobrescrevendo histórico remoto.
  • --tags: Envia todas as tags para o repositório remoto.
  • --all: Envia todas as branches locais.
  • --delete: Remove uma branch ou tag no remoto.
  • --dry-run: Mostra o que aconteceria no push, sem enviar de fato.
  • --atomic: Se várias refs falharem, cancela todo o push.
  • --mirror: Espelha completamente o repositório.
  • -q ou --quiet: Modo silencioso.
  • -v ou --verbose: Modo detalhado.

Boas Práticas Gerais

  • Commits pequenos e frequentes: facilitam rastreamento e reversão.
  • Nomes de branches e commits claros: use convenções como feature/login.
  • Pull Requests e Code Review: revisões antes de mesclar ajudam a qualidade.
  • Testes automatizados (CI): rodar testes em cada push/PR.
  • Taggear versões: facilita rollback e referência.
  • Não forçar push em branches compartilhadas: pode confundir o time.
  • Documentar o fluxo: descreva em README/Wiki como criar branches etc.
